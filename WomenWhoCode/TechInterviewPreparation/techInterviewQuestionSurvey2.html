Survey of programming interview questions: design patterns, web architecture, databases

By Elze Hamilton / @elze

Women Who Code technical interview preparation series

Introduction

This is a sample of interview questions that you might get about software design patterns, software architecture, and databases, if you are looking for an application developer or a web developer position.

If you work in a specialized field of computing, such as systems programming, firmware, drivers, scientific computing, data science, or other specialization, these questions might not reflect what you can expect an an interview.

In many programming interviews, writing code to solve a problem (on a whiteboard or a computer) is only a small part of the interview. Algorithms are often only a small part. The bulk of the interview will probably be answering questions about specific features of the programming language requiring for the job, writing code that demonstrates your knowledge of that language, also answering questions about software engineering, architecture, design, and best practices.

The purpose of this presentation

All of the questions here are only starting points for your study. The answers provided here are by no means complete or exhaustive. Rather, they point you in the direction you should look into for understanding the given topic.
It will be useful to work through the code examples provided here, and in the websites that are linked to in this presentation. In addition to that, try to come up with your own code samples that illustrate these problems.

Design patterns

"In software engineering, a design pattern is a general repeatable solution to a commonly occurring problem in software design."

Some design patterns are specific to programming languages, but many can be implemented in many languages.

Common Design patterns in object-oriented languages

  • Factory: the constructor of the class is private; to get an object of a class, you have to call a public static method that calls the constructor and returns the object;
  • Singleton: similar to Factory, but only one instance of the class is allowed to exist. The public static method always returns the same existing object.
  • Decorator: a method in the derived class "decorates" the method of the same name in the base class with new behavior.
  • Observer: an object (observable) calls its observers -- other objects that have subscribed to its notifications -- to notify them of the changes in itself. You can think of event handlers in Javascript (onclick, onmouseover, etc.) as observers, because the observable (e.g. a link, a button) "calls" them to notify that its state has changed, although that's an oversimplification.
  • ... and many more.

You should be able to explain a few design patterns and write code snippets illustrating them, e.g. coffee drink example for the Decorator pattern (can be found by googling).

Design patterns in your chosen language(s) and framework(s)

A good way to understand design patterns is to learn how they are implemented in the classes of frameworks used in your chosen language.

  • In .NET framework Stream class and its derived classes implement the Decorator pattern;
  • Angular.js uses the Singleton pattern to create the services that it injects. The injector creates each service only once.

More design patterns "in the wild"

System.Net.WebRequest class and its derived classes implement the Factory pattern. To create a request, call the Create method and pass in a URI. The Create method itself determines the appropriate protocol, such as FTP or HTTP, based on the URI and returns the appropriate subclass of WebRequest: HttpWebRequest, FtpWebRequest, or FileWebRequest. The caller doesn't need to know the specifics of each protocol, only how to invoke the factory and work with the WebRequest that gets returned.


WebRequest httpRequest = WebRequest.Create("http://foo.com");
WebRequest ftpRequest = WebRequest.Create("ftp://foo.com");

The actual type of httpRequest is HttpWebRequest, and the actual type of ftpRequest is FtpWebRequest, but because the constructors for those classes are private or protected, you can't instantiate them like this:


HttpWebRequest httpRequest = new HttpWebRequest("http://foo.com");
FtpWebRequest ftpRequest = new FtpWebRequest("ftp://foo.com");

DAO (Data Access Object) pattern

Data Access Object is a class that "wraps" a SQL table in code. Sometimes it's not just a SQL table, but a join of several tables, or generally results of some query.

It provides a lot of the same operations that you can do with a database table -- select from it, insert, update, etc. -- only it provides them in code. So instead of calling the database directly, you can call the corresponding methods off of the object.


public interface StudentDao {
   public List getAllStudents();
   public Student getStudent(int rollNo);
   public void updateStudent(Student student);
   public void deleteStudent(Student student);
}

The methods in the StudentDao are CRUD operations that roughly correspond to the SQL statements.

  • getAllStudents and getStudent() correspond to SELECT
  • updateStudent corresponds to UPDATE
  • deleteStudent corresponds to DELETE

getAllStudents should probably be overloaded to also take a SQL query as an argument. We rarely want to select all students. We usually want to select some subset of students based on the criteria in the WHERE clause.

Object-relational mapping frameworks

Another way to access and manipulate data in the code are Object-Relational Mapping frameworks (ORMs). They also "wrap" calls to the database in code, but in a different way.

Find out what ORMs are commonly used in your preferred language. E.g. C#: Entity Framework, NHibernate; Java: Hibernate; Ruby: ActiveRecord; Python: SQLAlchemy. Study that ORM and write a sample application that uses it.

Common ORM questions

  • What is lazy loading?
  • What is the opposite of lazy loading?
  • What are the advantages and gotchas of lazy loading?

Web architecture

What is MVC? What do Model, View, and Controller do?
Given a web framework (that you know, or that's most relevant to this position), such as .NET MVC or Ruby on Rails, describe what happens when a user's request hits the web server. What gets executed and in what order?

E.g. some function is invoked first, and determines which route matches the user request. Then, based on the route, the right method off of a right controller is called, etc., and in the end the right View function is called that creates the HTML of the resulting web page, and the page is sent to the user's browser.

REST

What does REST architecture mean?
What HTTP requests are used in REST? How are they different from one another?
What is a structure of a RESTful route?

RESTful routes should have nouns, not verbs, in them. The noun should be the model this route is operating on. For example, a route to retrieve a list of all users should be http://<app_url>/users, not http://<app_url>/get_all.

What should be a RESTful route to get a user by ID?

http://<app_url>/user/:id

What should be a RESTful route to modify a user by ID? Delete a user by ID?

They both should still be http://<app_url>/user/:id. The data, such as the new values of the user's fields, should be passed in the body of the request.

Exercise to the reader: what should be the route to create a new user?

Web security

Name two common front-end security threats.

Cross-site scripting (XSS), and cross-site request forgery (CSRF or XSRF). Be able to explain what they are, and what are common methods of protection against them.

A call to a web service is just an HTTP request, so without proper protection anyone from anywhere on the internet could call them. How do you make it so your web services are restricted only to the authenticated users?

Find out about various currently used ways of authenticating a user to web services. For example, the server could send a certain token to the frontend; frontend then would put an XAuth header with that token in every request it sends to the server. The webservice would expect that token to be present and to match the token it has, otherwise it would not process the request.

Databases and SQL

SQL databases are still very much in use at most software companies. Predictions of NoSQL databases replacing SQL have not come true, and are unlikely to come true in the forseeable future. Most software companies require some knowledge of SQL, at least for full stack or server-side development positions.

Common SQL questions

  • Queries: SELECT, INSERT, UPDATE, DELETE. WHERE clauses.
  • Most commonly used aggregate functions: MIN, MAX, SUM, COUNT
  • Clauses used with aggregate functions: GROUP BY and HAVING
  • Inner and outer joins. Be able to explain what they are. Be able to solve problems requiring outer joins.
  • Subqueries, e.g. SELECT x ... WHERE x IN (SELECT y FROM ...)

The best way to develop SQL knowledge, like any programming knowledge, is to solve exercises.

A good source for SQL exercises is http://sqlzoo.net. The exercises start simple and increase in difficulty. The only drawback is that while they show you the correct result, they won't show the correct solution.

Another good source is https://en.wikibooks.org/wiki/SQL_Exercises. It has fewer exercises, but provides solutions.

What are primary keys? Foreign keys? What are foreign key constraints?

What is an index? What is it useful for? What kinds of indices are there?

How is DELETE FROM <table> different from TRUNCATE <table>? If you wanted to delete all the data in a table, which one should you use and why?

Answer: parametrize the query. Know how to write a call a parametrized SQL query from your preferred programming language.

Parametrizing a SQL query

Let's say you have to construct a SQL query and pass these dates to its WHERE clause: 


string beginDateStr = "1996-07-31";
string endDateStr = "1998-08-01";

Don't do this -- don't concatenate arguments into the query:


var dbCommand = connection.CreateCommand();
dbCommand.CommandText = "SELECT name, department FROM students WHERE birthdate > " + beginDateStr + " AND birthdate <= " + endDateStr ;

Do this instead -- parametrize the query. (This is C#, but your preferred server-side programming language has something like that too.)


var dbCommand = connection.CreateCommand();
dbCommand.CommandText = "SELECT name, department FROM students WHERE birthdate > @beginDate AND birthdate <= @endDate" ;
SqlParameter beginDate = new SqlParameter("@beginDate", SqlDbType.DateTime);
beginDate.Value = DateTime.Parse(beginDateStr);
dbCommand.Parameters.Add(beginDate);
SqlParameter endDate = new SqlParameter("@endDate", SqlDbType.DateTime);
endDate.Value = DateTime.Parse(endDateStr);
dbCommand.Parameters.Add(endDate);

Primary and foreign keys

What should be the primary keys in these tables? What should be the foreign keys?

Primary key should uniquely identify every row. On all three tables it should be id.

Foreign keys assure that a field appears in table A only if there is a row with it as the primary key in table B.

id in products table should be the foreign key product_id in the shipments table.

id in stores table should be the foreign key store_id in the shipments table.
