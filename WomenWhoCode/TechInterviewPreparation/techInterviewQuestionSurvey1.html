Hashtables

also known as: associative arrays, dictionaries, hashmaps, key-value maps, lookup tables, maps

It's a collection of key-value pairs. Some keys may not have any values.

KeyValue
DogWoof
CatMeow
Fox
BirdChirp

In C#: IDictionary<type1, type2>, Hashtable (older)

In Java: HashTable, HashMap

In Python: Dictionary

In PHP: array (the kind with keys and values)

In Javascript: object (the object's fields are keys, field values are values)

Important things to know about dictionaries:

The key-value pairs are NOT sorted by key or by value. If you iterate through the keys, keep in mind that they'll be in any random order. If you want them to be in order, sort them first.

Lookup time for an element is O(1): constant-time independent of dictionary size.

This is made possible by the hash function. For every key it computes some value, e.g. hash("foo") = 3. That way the dictionary knows to always look for "foo" in slot 3. (It's an oversimplifation.)

Javascript questions 2. Variable scopes

What is a global variable in Javascript?

A variable whose scope is the window the script runs in.

How do you make variable global?

If you don't declare it with var, it will be global.

What is a local variable?

A variable declared with a var keyword is local to the scope it is declared in, such as a function.

Are global variables good? Why or why not?

They should be avoided. A global variable is easy to modify accidentally. Always declare the variables in the scope you need them in.

Javascript questions 3.

What is the difference between == and === operators?

== operator only checks the values of the operands, but === also checks types.


	var a1 = 5;
	var a2 = "5";
	console.log(a1 == a2); // true;
	console.log(a1 === a2); // false;
What does this keyword mean?

this refers to an object that can be different things depending on the context.

this as the global object (window)

console.log(this === window); // true

console.log(this.document === document); // true

Javascript questions 4. this inside a function

Constructor for Employee object: 


function Employee(empName, empDept, empjobTitle) {
 this.name = empName;
 this.department = empDept,
 this.jobTitle = empjobTitle;
 console.log(this.name + ", " + this.department + " " + this.jobTitle);
}
var emp1 = Employee("Foo", "Engineering", "Developer");
console.log(this.name + ", " + this.department + " " + this.jobTitle);

Let's call it as just a function, not a constructor:

var emp1 = Employee("Foo", "Engineering", "Developer")

It will print "Foo, Engineering, Developer"

What if we call the same console.log statement outside the function?

console.log(this.name + ", " + this.department + " " + this.jobTitle);

It prints the same: "Foo, Engineering, Developer"

That tells us that this inside Employee() function was the global this (window). 


function Employee(empName, empDept, empjobTitle) {
 this.name = empName;
 this.department = empDept,
 this.jobTitle = empjobTitle;
 console.log(this.name + ", " + this.department + " " + this.jobTitle);
}
var emp2 = Employee("Bar", "Implementations", "Implementations Engineer");
console.log(this.name + ", " + this.department + " " + this.jobTitle);

Now let's call Employee() as a constructor:

var emp2 = new Employee("Bar", "Implementations", "Implementations Engineer")

The difference is, we are calling it with a new keyword, so it constructs an Employee object.

It will print "Bar, Implementations, Implementations Engineer"

What if we call the same console.log statement outside the function?

console.log(this.name + ", " + this.department + " " + this.jobTitle);

It prints, as before "Foo, Engineering, Developer"

this inside the Employee object is now different from the global this.

Constructing an Employee object caused this inside the object to refer to the object itself.

Design patterns

"In software engineering, a design pattern is a general repeatable solution to a commonly occurring problem in software design."

Some design patterns are specific to programming languages, but many can be implemented in many languages.

Common Design patterns in object-oriented languages

  • Factory: the constructor of the class is private; to get an object of a class, you have to call a public static function that calls the constructor and returns the object;
  • Singleton: similar to Factory, but only one instance of the class is allowed to exist. The public static function always returns the same existing object.
  • Decorator: a method in the derived class "decorates" the method of the same name in the base class with new behavior.
  • Observer: an object (observable) calls its observers -- other objects that have subscribed to its notifications -- to notify them of the changes in itself.
  • ... and many more.

Design patterns in your chosen language(s) and framework(s)

A good way to understand design patterns is to learn how they are implemented in the classes of frameworks used in your chosen language.

  • In .NET framework Stream class and its derived classes implement the Decorator pattern;
  • Angular.js uses the Singleton pattern to create the services that it injects. The injector creates each service only once.

More design patterns "in the wild"

System.Net.WebRequest class and its derived classes implement the Factory pattern. To create a request, call the Create method and pass in a URI. The Create method itself determines the appropriate protocol, such as FTP or HTTP, based on the URI and returns the appropriate subclass of WebRequest: HttpWebRequest, FtpWebRequest, or FileWebRequest. The caller doesn't need to know the specifics of each protocol, only how to invoke the factory and work with the WebRequest that gets returned.


WebRequest httpRequest = WebRequest.Create("http://foo.com");
WebRequest ftpRequest = WebRequest.Create("ftp://foo.com");

The actual type of httpRequest is HttpWebRequest, and the actual type of ftpRequest is FtpWebRequest, but because the constructors for those classes are private or protected, you can't instantiate them like this:


HttpWebRequest httpRequest = new HttpWebRequest("http://foo.com");
FtpWebRequest ftpRequest = new FtpWebRequest("ftp://foo.com");

DAO (Data Access Object) pattern

Data Access Object is a class that "wraps" a SQL table in code. Sometimes it's not just a SQL table, but a join of severa tables, or generally results of some query.

It provides a lot of the same operations that you can do with a database table -- select from it, insert, update, etc. -- only it provides them in code. So instead of calling the database directly, you can call the corresponding methods off of the object.


public interface StudentDao {
   public List getAllStudents();
   public Student getStudent(int rollNo);
   public void updateStudent(Student student);
   public void deleteStudent(Student student);
}

The methods in the StudentDao are CRUD operations that roughly correspond to the SQL statements.

  • getAllStudents and getStudent() correspond to SELECT
  • updateStudent and getStudent() correspond to UPDATE
  • deleteStudent and getStudent() correspond to DELETE

getAllStudents should probably be overloaded to also take a SQL query as an argument. We rarely want to select all students. We usually want to select some subset of students based on the criteria in the WHERE clause.

Databases and SQL

SQL databases are still very much in use at most software companies. Predictions of NoSQL databases replacing SQL have not come true, and are unlikely to come true in the forseeable future. Most software companies require some knowledge of SQL, at least for full stack or server-side development positions.

Common SQL questions

  • Queries: SELECT, INSERT, UPDATE, DELETE. WHERE clauses.
  • Most commonly used aggregate functions: MIN, MAX, SUM, COUNT
  • Clauses used with aggregate functions: GROUP BY and HAVING
  • Inner and outer joins. Be able to explain what they are. Be able to solve problems requiring outer joins.
  • Subqueries, e.g. SELECT x ... WHERE x IN (SELECT y FROM ...)

The best way to develop SQL knowledge, like any programming knowledge, is to solve exercises.

A good source for SQL exercises is http://sqlzoo.net. The exercises start simple and increase in difficulty. The only drawback is that while they show you the correct result, they won't show the correct solution.

Another good source is https://en.wikibooks.org/wiki/SQL_Exercises. It has fewer exercises, but provides solutions.

Time complexity of the anagrams-the-naive-way algorithm

It is O(n2) .

Why?

The outer loop iterates through the first word.

For each iteration of the outer loop, the inner loop traverses the whole second word.

Always think of the time complexity of the solution

Interviewers will probably ask about it.

Easy shortcuts to estimate time complexity:

  • Searching for an element in an ordered list or binary tree is O(log(n))
  • Sorting a list of elements in most general case (if nothing special is known about the elements) is O(n log(n))

Identify operations in your algorithm that are executed independently of each other. Take a maximum of their time complexity.

E.g. in the anagrams solution there are 3 standalone loops of complexities O(n), O(n), and O(1) => total is O(n).

If an operation involves a suboperation (e.g. doing something in each iteration of the loop), multiply complexity of the "outer" operation by the complexity of the "inner" operation.

E.g. in the "naive anagrams" algorithm, the outer loop is O(n) and the inner loop is O(n) => total is O(n2)
