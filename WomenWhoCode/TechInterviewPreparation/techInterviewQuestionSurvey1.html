Survey of programming interview questions: Javascript and object-oriented languages

By Elze Hamilton / @elze

Women Who Code technical interview preparation series

Introduction

This is a sample of interview questions that you might get about Javascript and some object-oriented languages, if you are looking for an application developer or a web developer position.

If you work in a specialized field of computing, such as systems programming, firmware, drivers, scientific computing, data science, or other specialization, these questions might not reflect what you can expect an an interview.

In many programming interviews, writing code to solve a problem (on a whiteboard or a computer) is only a small part of the interview. Algorithms are often only a small part. The bulk of the interview will probably be answering questions about specific features of the programming language requiring for the job, writing code that demonstrates your knowledge of that language, also answering questions about software engineering, architecture, design, and best practices.

The purpose of this presentation

All of the questions here are only starting points for your study. The answers provided here are by no means complete or exhaustive. Rather, they point you in the direction you should look into for understanding the given topic.
It will be useful to work through the code examples provided here, and in the websites that are linked to in this presentation. In addition to that, try to come up with your own code samples that illustrate these problems.

More and more development positions are full-stack, and they require some knowledge of Javascript. If the position you are applying to is for developing an application that has a web front, chances are the employer will expect you to know at least some Javascript.

So unless you are applying for positions that are strictly server-side development (there aren't many of those), it's good to know Javascript.

Many times the interviewer will show you code snippets, and ask what the snippet would print, like in the following pages of this presentation.

Null is a value, but undefined is a type.

var pet = new Pet("Fluffy")

Javascript questions 2. Variable scopes

What is a global variable in Javascript?

A variable whose scope is the window the script runs in.

How do you make variable global?

If you don't declare it with var, it will be global.

What is a local variable?

A variable declared with a var keyword is local to the scope it is declared in, such as a function.

Are global variables good? Why or why not?

They should be avoided. A global variable is easy to modify accidentally. When all parts of your code are modifying the same variable, it becomes difficult to reason about what its value should be at a given point. Always declare the variables in the scope you need them in.

Javascript questions 3.

What is the difference between == and === operators?

== operator only checks the values of the operands, but === also checks types.


	var a1 = 5;
	var a2 = "5";
	console.log(a1 == a2); // true;
	console.log(a1 === a2); // false;
What does this keyword mean?

this refers to an object that can be different things depending on the context.

this as the global object (window)

console.log(this === window); // true

console.log(this.document === document); // true

Javascript questions 4. this inside a function

Constructor for Employee object: 


function Employee(empName, empDept, empjobTitle) {
 this.name = empName;
 this.department = empDept;
 this.jobTitle = empjobTitle;
 console.log(this.name + ", " + this.department + " " + this.jobTitle);
}
var emp1 = Employee("Foo", "Engineering", "Developer");
console.log(this.name + ", " + this.department + " " + this.jobTitle);

Let's call it as just a function, not a constructor:

var emp1 = Employee("Foo", "Engineering", "Developer")

It will print "Foo, Engineering, Developer"

What if we call the same console.log statement outside the function?

console.log(this.name + ", " + this.department + " " + this.jobTitle);

It prints the same: "Foo, Engineering, Developer"

That tells us that this inside Employee() function was the global this (window). 


function Employee(empName, empDept, empjobTitle) {
 this.name = empName;
 this.department = empDept,
 this.jobTitle = empjobTitle;
 console.log(this.name + ", " + this.department + " " + this.jobTitle);
}
var emp2 = Employee("Baz", "Implementations", "Implementations Engineer");
console.log(this.name + ", " + this.department + " " + this.jobTitle);

Now let's call Employee() as a constructor:

var emp2 = new Employee("Baz", "Implementations", "Implementations Engineer")

The difference is, we are calling it with a new keyword, so it constructs an Employee object.

It will print "Baz, Implementations, Implementations Engineer"

What if we call the same console.log statement outside the function?

console.log(this.name + ", " + this.department + " " + this.jobTitle);

It prints, as before "Foo, Engineering, Developer"

this inside the Employee object is now different from the global this.

Constructing an Employee object caused this inside the object to refer to the object itself.

Javascript questions 5. bind, call, apply

A general rule is that this in the context of an object (e.g. if a function is called off of an object) will refer to the object itself, while outside of any object it will be the "global" this (usually the browser window). But there are many more nuances in the use of this. Be sure to study them all. A good reference is https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/this.

As we see, this can refer to different objects, and it can be confusing.

Javascript questions 6. More Javascript development questions

What kind of inheritance model Javascript uses?

Prototypal / prototype-based inheritance.

How is it different from class-based inheritance?
Does Javascript have private variables (like many class-based languages)?

No, it doesn't.

Can you emulate private fields and methods in Javascript?

Yes, you can. One way to do it would be with Module Pattern: https://toddmotto.com/mastering-the-module-pattern using IIFE (immediately-invoked function expression) and closures. Study this article to understand uses of closures and IIFE.

How do you create namespaces in Javascript?

To keep the variable names from colliding (especially if your Javascript code has to coexist with third-party Javascript libraries) you can create a namespace for your code. Many ways of creating namespaces use IIFE. You can read about namespacing patterns here: https://addyosmani.com/blog/essential-js-namespacing

Exception handling

All the commonly used programming languages have exceptions. Best practices of exception handling are common interview questions.

Usually exception-handling constructs are something like that:


try {
	... some code that can possibly throw an exception
}
catch (ExceptionType e) {
	... do something with the exception
}
catch (AnotherExceptionType e) {
	... do something with the exception
}
...
catch (YetAnotherExceptionType e) {
	... do something with the exception
}
finally {
	... some code that gets executed whether or not exception was thrown
}

The keywords may be different in different languages. For example, Python has try ... except ... else , Ruby has rescue instead of catch, etc.

Exception handling, continued

A common interview question: if the code in the catch block throws an exception, will the finally block be executed?

The way to find out is to write a piece of code for this scenario in your language.

This scenario describes a common situation: if an exception occurs in the try block, the catch block often logs it to a file or a database. What if it can't open the database connection, or the log file doesn't exist? Then an exception will be thrown in the catch block.

    Other exception handling principles:

    Exceptions should be treated as rare, because handling them is computationally costly. Don't use exceptions to handle commonly-occuring scenarios. For example, if a function often returns null, don't make the null value throw an exception. Rather, design your code so that null would be an acceptable value.

    Another example. Your regularly-running script deletes files. Some files' permissions don't let your script delete them. This happens often, and you don't want your script to throw an exception when this occurs. You want it to check if the file's permissions allow deletion, and if not, move on to the next file. However, if your script discovers that the whole directory is missing, it should definitely throw an exception, because it can't work with a nonexistent directory.
